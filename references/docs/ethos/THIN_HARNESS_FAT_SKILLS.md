# 瘦线束，胖技能 (Thin Harness, Fat Skills)

Steve Yegge 说，使用 AI 编码代理的人“生产力是当今使用 Cursor 和聊天工具的工程师的 10 到 100 倍，大约是 2005 年 Google 员工的 1000 倍。”

这是一个真实的数字。我见过。我亲历过。但是当人们听到 100 倍时，他们想到的是：更好的模型。更聪明的 Claude。更多的参数。

这完全是错误的思考框架。实现 2 倍和 100 倍生产力的人使用的是同一个模型。区别在于索引卡上的五个概念。

## 线束 (The harness) 才是秘密武器

2026 年 3 月 31 日，Anthropic 意外地将 Claude Code 的整个源代码发布到了 npm 注册表中。512,000 行代码。当我阅读它时，它证实了我在 YC 教授的一切。秘诀不是模型。而是包裹模型的东西：线束 (the harness)。实时仓库上下文、提示词缓存、专用工具、上下文膨胀最小化、结构化会话内存、并行子代理。

这些都不是为了让模型变得更聪明。这些都是为了在完全正确的时刻为模型提供完全正确的上下文，而不是让它淹没在噪音中。

这是唯一重要的问题。答案有一个特定的形状。我称之为**瘦线束，胖技能 (Thin harness, fat skills)**。

## 五个定义

瓶颈从来不是模型的智能。瓶颈在于模型不理解你的架构。模型已经知道如何推理、综合和编写代码。它们失败是因为它们不了解你的数据。五个定义解决了这个问题。

### 定义 1：技能文件 (The skill file)

技能文件是一个可重用的 markdown 程序，它教模型如何做某事。不是“做什么”。用户提供具体内容。技能提供过程。

**Markdown 实际上是代码。** 技能文件是对能力的一种比僵硬的源代码更完美的封装，因为它用模型本来就用来思考的语言描述了过程、判断和上下文。

左边是一个名为 `/investigate` 的技能。七个步骤：确定数据集范围、建立时间线、对每个文档进行分类、综合、论证双方观点、引用来源。它接受三个参数：TARGET（目标）、QUESTION（问题）和 DATASET（数据集）。

右边是对同一个技能的两次完全不同的调用。一次是指向 Dr. Sarah Chen 和 210 万封发现阶段的电子邮件，询问安全科学家是否被压制。另一次是指向 Pacific Enterprise Services 和联邦选举委员会的备案文件，询问一家空壳公司是否在协调竞选捐款。

同一个技能。同样的七个步骤。同一个 markdown 文件。在一个案例中，它是一个医学研究分析师。在另一个案例中，它是一个法务调查员。技能描述了判断的过程。调用提供了世界背景。

**这是大多数人错过的一个关键洞见：技能文件就像方法调用一样运作。** 它接受参数。你可以用不同的参数调用它。根据你传入的内容，相同的过程会产生完全不同的能力。这不是提示词工程。这是软件设计，使用 markdown 作为编程语言，使用人类判断作为运行时。

### 定义 2：线束 (The harness)

线束是运行 LLM 的程序。它做四件事：在循环中运行模型、读写文件、管理上下文、强制执行安全边界。它应该保持“瘦”。

反模式是厚工具和瘦技能：占据一半上下文窗口的 40 多个工具定义。“上帝工具”有 2 到 5 秒的 MCP 往返时间。将每个端点变成一个工具的 REST API 包装器。3 倍的 token，3 倍的延迟，3 倍的失败率。

你应该构建的是：一个能在 100 毫秒内完成任何浏览器操作的 Playwright CLI。对比一下：Chrome MCP 需要 15 秒来截屏 + 查找 + 点击 + 等待 + 读取。Playwright CLI 只需要 200 毫秒来截屏 + 断言。速度快了 75 倍。软件不需要再那么精贵了。准确构建你所需要的东西。

### 定义 3：解析器 (The resolver)

解析器是上下文的路由表。当任务类型 X 出现时，首先加载文档 Y。

技能说明如何做。解析器规定何时加载什么。开发人员更改一个提示词。如果没有解析器，他们直接发布。有了解析器，模型首先读取 `docs/EVALS.md`，其中写道：运行评估套件，比较分数，如果准确率下降超过 2%，则还原并调查。开发人员不知道评估套件的存在。解析器在正确的时刻加载了正确的上下文。

Claude Code 有一个内置的解析器。每个技能都有一个描述字段，模型会自动将用户意图与技能描述进行匹配。你永远不需要记住 `/ship` 的存在。描述就是解析器。这就像 Clippy。只是它真的有用。

坦白说：我的 `CLAUDE.md` 有 20,000 行。我遇到的每一件事都在那里。每一个怪癖，每一个模式，每一个教训。这太荒谬了。模型的注意力下降了。Claude Code 甚至告诉我要删减它。修复方法：大约 200 行。只是指向文档的指针。解析器会在相关的时候加载正确的文档。

### 定义 4：潜在与确定性 (Latent vs. Deterministic)

系统中的每一步都属于其中之一。

**潜在空间 (Latent space)** 是智能存在的地方。模型进行阅读、解释、决策。判断。综合。模式识别。

**确定性 (Deterministic)** 是信任存在的地方。相同的输入，相同的输出。每次都一样。SQL。代码。数学运算。

LLM 可以在晚宴桌上安排 8 个人入座。如果你让它安排 800 个座位，它会幻觉出一张看起来合理但完全错误的座位表。那是一个被强行推入潜在空间的确定性问题。最糟糕的系统把错误的工作放在了错误的一边。

### 定义 5：二值化 (Binarization)

模型阅读关于一个主题的所有内容，并写出一份结构化的档案。阅读 50 份文档，产出一页纸的判断。

没有任何 SQL 查询能产生这个结果。没有任何 RAG 流水线能产生这个结果。模型必须实际阅读，在脑海中保留矛盾，注意什么改变了以及何时改变的，并写出结构化的情报。这就是 AI 对真正的知识工作有用的原因。

## 架构

三层：

顶层是**胖技能 (Fat skills)**。编码判断、过程和领域知识的 Markdown 程序。这是 90% 价值存在的地方。

中间是**瘦 CLI 线束 (Thin CLI harness)**。大约 200 行。JSON 输入，文本输出。默认为只读。先用 CLI，然后再添加 MCP。

底层是**你的应用 (Your app)**。查询数据库。读取文档。搜索。时间线。确定性基础。

将智能向上推入技能中。将执行向下推入确定性工具中。保持线束薄弱。

## 学习系统：YC 创业学校 (YC Startup School)

让我向你展示所有五个定义是如何协同工作的。不是在理论上。而是在我们在 YC 构建的一个真实系统中。

大通中心。2026 年 7 月。6,000 名创始人。每个人都有结构化的申请、问卷答案、1:1 顾问聊天记录和公共信号：X 帖子、GitHub 提交、显示其交付速度的 Claude Code 日志。

传统方法：一个 15 人的项目团队阅读申请，凭直觉做出判断，更新电子表格。这对 200 名创始人有效。在 6,000 人时就会崩溃。

没有人能在工作记忆中保留 6,000 份档案，并注意到 AI 代理基础设施队列的最佳三名候选人是拉各斯的一位开发工具创始人、新加坡的一位合规创始人，以及布鲁克林的一位 CLI 工具创始人——他们在 1:1 聊天中都用不同的词语描述了相同的痛点。

模型可以。

**第 1 步：丰富每一位创始人。**
`/enrich-founder` 技能：提取所有来源，运行丰富流水线，记录日记，突出显示他们所说的与他们实际构建的内容。调用右侧的确定性工具：查找陈旧档案的 SQL、GitHub 统计信息、演示 URL 的浏览器测试、社交信号提取、用于公司情报的 CrustData。
Cron 每晚凌晨 2 点运行。6,000 份档案，每晚更新，始终保持新鲜。

二值化输出捕捉到了关键字搜索找不到的东西：

```
FOUNDER: Maria Santos
COMPANY: Contrail (contrail.dev)
SAYS: "Datadog for AI agents"
ACTUALLY BUILDING: 80% of commits are in billing module.
  She's building a FinOps tool disguised as observability.
```

“所说”与“实际构建”。这需要阅读 GitHub 提交历史、申请和顾问记录，并同时将这三者保留在脑海中。

**第 2 步：匹配 6,000 名创始人。做出判断。**
这就是“技能即方法调用”真正发挥作用的地方。三个调用：
`/match-breakout`：1,200 名创始人，按行业亲和力分组，每组 30 人。嵌入 + 确定性分配。
`/match-lunch`：600 名创始人，机缘匹配（跨界），每桌 8 人，不重复。LLM 发明主题，然后分配。
`/match-live`：寻找该区域的任何人，最近邻嵌入，200 毫秒实时响应，1:1 配对，尚未见过面的人。

同一个技能。三次调用。三种完全不同的匹配策略。不同的参数，不同的策略，不同的群体规模。技能描述了过程。参数决定了输出。

模型的判断是：“Santos 和 Oram 都是做 AI 基础设施的，但他们不是竞争对手。Santos 是做成本归因的，Oram 是做编排的。把他们放在同一组。” 以及：“Kim 申请的是‘开发工具’，但他的 1:1 记录显示他正在构建 SOC2 合规自动化。把他移到金融科技/监管科技。”

没有嵌入能捕捉到 Kim 的重新分类。没有算法能做到这一点。模型必须阅读整个档案。

**第 3 步：自学习循环。**
活动结束后，`/improve` 技能读取 NPS 调查，对 "OK" 响应（不是 "bad"，也不是 "stellar"）进行分类，并提取模式。然后它提出新规则，并将其写回匹配技能中：

```
When attendee says "AI infrastructure"
    but startup is 80%+ billing code:
    -> Classify as FinTech, not AI Infra.

When two attendees in same group
    already know each other:
    -> Penalize proximity.
       Prioritize novel introductions.
```

这些规则被写回技能文件。下一次运行会自动使用它们。技能重写了它自己。

7 月的活动：12% 的 "OK" 评分。下一次活动：4%。技能文件学习了 "OK" 实际意味着什么。

其他领域的模式也是相同的：检索、阅读、分类、计数、综合。然后：调查、探究、记录、重写技能。这适用于任何地方。

## OpenClaw：技能的安身之所

我想告诉你另一种线束。不是用于编码的。而是用于其他一切的。

我在 OpenClaw 上运行我的个人 AI 代理。它有一个角色设定，知道我是谁，并维护着一个包含数千个相互链接的文件的知识库。但让它起作用的原理是完全相同的。瘦线束，胖技能。

几天前我在推特上发过这个：
> *不允许你做一次性的工作。如果我要求你做某事，并且这种事情需要再次发生，你必须：第一次在一个包含 3 到 10 个项目的小批次上手动执行。向我展示输出。如果我批准，将其编纂成技能文件。如果它应该自动运行，把它放到 cron 上。*
> *测试：如果我必须向你要求某事两次，你就失败了。*

这引起了共鸣：一千个点赞，两千五百个书签。人们认为这是一个提示词工程技巧。并不是。这是相同的架构。

Claude Code 是用于编码的最佳工具。OpenClaw 是用于其他一切的最佳工具：电子邮件、日历、会议、人员、研究、警报。相同的原则。瘦线束。胖技能。技能就是方法调用。参数发生改变。过程保持不变。

我编写的每一个技能都是一次永久的升级。它永远不会退化。它永远不会忘记。它在凌晨 3 点我睡觉时运行。当下个模型发布时，每个技能都会立即变得更好，因为在潜在步骤中的判断力提高了，而确定性步骤仍然完全可靠。

这就是 Yegge 所说的 10 倍到 100 倍的方法。不是一个更聪明的模型。而是胖技能，瘦线束，以及将一切编纂成典的纪律。

系统在不断复合。构建一次。它永远运行。

五个定义。三层。一个原则。瘦线束，胖技能。

## 代理决策指南：技能还是代码？

在构建 GBrain 功能时，使用此决策指南：

| 问题 | 如果是 | 如果否 |
|----------|--------|--------|
| 代理是否需要思考、适应或提出问题？ | **技能** (诀窍 markdown) | 代码 |
| 相同的输入是否总是产生相同的输出？ | **代码** (CLI 命令) | 技能 |
| 是否需要对用户的环境进行判断？ | **技能** | 代码 |
| 它是查找、列表还是状态检查？ | **代码** | 可能是技能 |
| 它是否根据对话上下文改变行为？ | **技能** | 代码 |

**GBrain 示例：**
- `gbrain integrations list` = **代码** （读取文件，检查环境变量，确定性的）
- `gbrain integrations status` = **代码** （检查环境变量 + 心跳，确定性的）
- `gbrain integrations doctor` = **代码** （运行健康检查，确定性的）
- `gbrain integrations stats` = **代码** （聚合 JSONL，确定性的）
- 诀窍设置流程 = **技能** （询问 API 密钥，适应环境，验证）
- 诀窍变更日志展示 = **技能** （代理以对话方式描述变更）
- 实体检测 = **技能** （读取消息，决定什么是重要的，创建页面）
- 会议摄取 = **技能** （读取记录，提取实体，更新页面）

**规则：** 如果它是查找表，它就是代码。如果代理需要思考，它就是技能。
