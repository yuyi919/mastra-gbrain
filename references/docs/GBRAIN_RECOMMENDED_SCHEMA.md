# Brain：法学硕士维护的知识库
针对任何想要建立和维护个人知识库的人工智能代理的系统提示。这描述了使其发挥作用的模式、架构和操作规则。
将其作为技能或系统提示放入代理的工作区中。您的代理人将建造其余的部分。
---

## 这是什么
一个个人智能系统，您的 AI 代理可以构建并维护一个相互链接的 wiki，其中包含您所了解的关于您的世界的所有内容（人员、公司、交易、项目、会议、想法），并将其作为结构化、交叉引用的 Markdown 文件。代理编写并维护所有这些。你指导、策划和思考。

关键见解：**知识管理已经失败了 30 年，因为维护工作落在了人类身上。 LLM 代理改变了这个方程式——他们不会感到无聊，不会忘记更新交叉引用，并且可以一次性处理 50 个文件。** 您的 wiki 保持活力，因为维护成本接近于零。
## 三个基本原则
### 1. 每条知识都有一个主要目录（MECE 目录）
每一条知识都经过决策树并准确地落在一个目录中。没有重复的页面，对内容的去向没有任何歧义。
这是最重要的结构性决策。没有它，知识库就会腐烂——同样的事实存在于三个不同的地方，具有三个不同的版本，没有人知道哪个是最新的，并且代理（或人类）不再信任系统。具有明确解析器规则的 MECE 目录可以防止这种情况发生。
每个目录都有一个“README.md”（解析器），它回答两个问题：
1. **这里发生了什么**——带有具体测试的积极定义
2. **此处不包含的内容** — 代理可能会混淆的与相邻目录的主要区别
大脑还有一个顶级“RESOLVER.md”——代理在归档任何内容时都会走的编号决策树。当两个目录看起来合适时，消歧规则就会打破平局。当没有合适的东西时，该项目就会进入“inbox/”——这本身就是模式需要发展的信号。
**代理必须在创建任何新页面之前读取解析器。**这不是可选的。
**重要的细微差别：MECE 适用于目录，而不是现实。** 真实的人和实体是多方面的 - 政治创始人也可以是朋友、捐助者、媒体演员和招聘候选人。解析器为其页面（人员/）选择“主要主页”，但页面本身使用类型化反向链接和交叉引用来显示其所有方面。 MECE 规则防止重复页面，而不是重复关系。交叉引用是在不违反每个实体一页规则的情况下保留邻接性的方式。
### 2.编译真相+时间线（两层页）
每个大脑页面都有两层，由水平线（`---`）分隔：
**在线之上 - 编译的真相。**始终是最新的，当新信息到达时总是重写。从一段执行摘要开始。如果你只读了这篇文章，你就知道比赛的状况。接下来是结构化状态字段、开放线程（活动项目 - 解决后删除）和另请参阅（交叉链接）。
**线下 - 时间线。**仅附加，从不重写。按时间倒序排列的证据日志。每个条目：日期、来源、发生的事情。当一个打开的线程得到解决时，它会带着它的解决方案移到这里。
如果有人问“目前情况如何？” - 阅读上面的内容。如果有人问“发生了什么事？” ——阅读下面的内容。顶部是当前摘要。最下面是源日志。
这是 Karpathy wiki 模式的杀手级功能：**综合是预先计算的。** 与 RAG 不同，在 RAG 中，LLM 在每次查询时都从头开始重新导出知识，而您的大脑已经完成了工作。交叉引用已经存在。矛盾已经被标记出来了。
### 3. 每个信号都会触发浓缩
每当任何信号触及个人或公司（会议、电子邮件、推文、日历事件、联系人同步、对话提及）时，丰富管道就会启动。大脑的成长是正常运作的副作用，而不是你记得要做的一项单独的任务。
这就是操作大脑与卡帕蒂研究维基的区别。他描述了如何摄取您手动添加的源。一个可操作的大脑更进一步——每个管道（会议、电子邮件、社交媒体、联系人）都会自动触发它所接触的每个实体的丰富化。您永远不必记住更新某人的页面。系统这样做是因为管道连接正确。
## 将其连接到您的代理中
大脑必须作为硬性规则而不是建议在代理配置（AGENTS.md 或同等内容）中引用。具体来说：
1. **在创建任何大脑页面之前 → 阅读 RESOLVER.md。** 这应该位于您的代理的操作规则中，而不是隐藏在文档中。
2. **在回答任何有关人员、公司、交易或战略的问题之前 → 首先搜索大脑。** 即使代理认为它知道答案。文件内容是最新的；特工对他们的记忆变得陈旧。
3. **丰富技能会在每个信号上触发。** 每个摄取路径（会议处理、电子邮件分类、社交监控、联系人同步）都应在遇到个人或公司时调用丰富管道。这是布线，而不是纪律。如果依赖于智能体记忆，它最终会被遗忘。
4. **更正是最高价值的数据。** 如果用户更正代理关于个人、公司、交易或决策的信息，它会立即写入大脑。没有批处理，没有延迟。
权限链： **代理配置（AGENTS.md）说“读取 RESOLVER.md” → RESOLVER.md 是决策树 → 每个目录 README.md 是本地解析器 → schema.md 定义页面结构 → 丰富技能定义丰富协议。**
＃＃ 建筑学
三层：
**原始来源** — 会议记录、电子邮件、推文、网络研究、API 响应、日历事件、联系数据。不变的。代理读取这些内容但从不修改它们。存储在 `sources/` 和 `.raw/` sidecar 目录中。
**大脑** — 相互链接的 Markdown 文件的目录。人员页面、公司页面、交易页面、会议页面、项目页面、概念页面。代理完全拥有这一层。它创建页面，在新信息到达时更新页面，维护交叉引用，并使一切保持一致。你读了它；代理写的。
**模式** — 一份文档（这个文档，加上“schema.md”和“RESOLVER.md”），告诉智能体大脑的结构、约定是什么以及要遵循的工作流程。这是关键的配置文件 - 它使您的代理成为纪律严明的知识维护者，而不是通用的聊天机器人。
## 数据库 + Markdown 架构
Markdown wiki 是面向人类的层——人类和法学硕士的主要界面。但这并不是事实的唯一来源。结构化数据库层提供了基础，并从中生成 Markdown。
### 四种数据库原语
**实体注册表** — 一张表中的规范 ID、所有别名、所有外部 ID（LinkedIn 会员 ID、X 用户 ID、电子邮件地址、电话号码）。这是“这是同一个人吗？”的唯一事实来源。当您合并两个实体时，这是一个数据库操作（将两个 ID 指向同一规范记录），而不是具有交叉引用修复的文件合并操作。
**事件分类账** - 每个触及大脑的信号都是一个不可变的事件：参加会议、收到电子邮件、发布推文、完成充实、应用用户校正。事件有来源：来源、时间戳、置信度、原始有效负载参考。 Markdown 页面的时间线部分是从此分类帐生成的。您永远不会因为页面重写出错而丢失事件。
**事实存储**——具有出处的结构化声明。 “Jane Doe 是 Acme 的首席技术官”，内容为“source=crustdata,confidence=high,observed_at=2026-04-07”。当两个消息来源不一致时（LinkedIn 称 CTO，公司网站称 VP Engineering），冲突就会表现为同一领域具有不同价值观的两个事实。该行上方的已编译真值部分是根据事实存储的最新置信值生成的。矛盾变成了数据，而不是错误。
**关系图** — 实体之间的类型化边。人→公司（角色：CTO，开始时间：2024-01），人→人（关系：共同创立公司），公司→交易（类型：A轮，日期：2025-03）。启用 markdown grep 无法回答的图形查询：“我知道谁投资了人工智能基础设施公司？”变成一种穿越，而不是祈祷。
### 为什么这很重要
- **身份解析**成为数据库操作（合并实体 ID），而不是手动交叉引用修复的文件合并操作
- **矛盾是结构性的**（同一领域和不同来源具有不同值的两个事实）而不是文本上的（希望法学硕士注意到散文中埋藏的差异）
- **并发性得到解决** - 事件附加到分类账，事实更新到存储，Markdown 被重建。共享文件不再发生合并冲突
- **图形查询有效** — “我在这家公司认识谁？”以及“这位投资者支持了哪些我也认识其创始人的公司？”成为数据库查询，而不是不可能的 grep 链
### 文件层约定
Markdown 层使用直接映射到数据库原语的约定：
1. **使用 frontmatter 获取结构化元数据** — 您想要查询的任何内容（角色、公司、阶段、分数、标签）都放在 YAML frontmatter 中，而不是隐藏在散文中。这些映射到事实存储。
2. **使用 `.raw/` 作为出处** — 保存每个 API 响应的来源和时间戳。这些映射到事实存储中的出处记录。
3. **将时间线视为事件流** — 注明日期、来源、仅附加。这些映射到事件分类帐。
4. **在概念上将编译后的真相与证据分开**——线以上是综合；线以下就是证据。综合是生成的视图；证据是可查询的记录。
5. **一致地使用规范的 slug** — 每个交叉引用都使用文件名 slug。这些是注册表中的实体 ID。

## 实体身份和重复数据删除
在由会议、电子邮件、社交媒体、联系人和 API 提供支持的系统中，**实体身份是第一个真正的失败模式。**如果没有规范的身份层，您最终将得到微妙的裂脑页面 — 会议记录中的“Jane Smith”、电子邮件中的“J. Smith”和 Twitter 中的“jsmith”，所有这些都为同一个人创建单独的页面。
### 规范蛞蝓
每个实体都有一个规范的 slug 作为其稳定 ID：
- People：`first-last.md`（全部小写，连字符用于空格）
- 公司：`公司名称.md`
- 如果发生冲突，请消除歧义：`david-liu-crustdata.md`、`david-liu-meta.md`
文件名就是身份。所有引用、交叉链接和 .raw/ sidecars 都使用这个 slug。
### 别名
人们在不同的来源中有很多名字。 frontmatter `aliases` 字段捕获所有已知的变体：
```yaml
aliases: ["Jenny Shao", "Jenny G. Shao", "JennyGShao", "jennifer.shao@company.com"]
```

别名包括：会议记录中的拼写错误、婚前姓名、昵称、电子邮件地址、社交账号和语音变体。当丰富技能遇到已知实体的新名称变体时，它会将变体添加到别名中 - 它不会创建新页面。
### 重复数据删除协议
在创建任何新页面之前，代理必须：
1.按名称搜索现有页面（精确和模糊）
2. 在所有页面搜索别名： `grep -rl "NAME_VARIANT" /data/brain/people/ --include="*.md"`
3. 检查 .raw/ sidecars 是否匹配电子邮件地址或社交账号
4. 如果找到匹配项 → 更新现有页面（如果名称变体是新的，则添加别名）
5. 如果不匹配→创建一个新页面
### 合并协议
当你发现两个页面是同一个人时：
1. 选择较完整的页面作为幸存者
2. 将所有时间线条目从重复项合并到幸存者中（按时间顺序）
3. 合并所有别名
4. 更新指向重复项的所有交叉引用
5.删除重复项
6. 提交消息：`merge: [duplicate] into [survivor]`
在每周 lint 期间，积极寻找潜在的重复项：相似的名称、同一家公司、不同页面上的相同电子邮件。
## 关键消歧规则
最常见的归档混乱以及解决方法：
- **概念与想法：** 你能将它作为一个框架“教授”吗？ → 概念。你能“建造”它吗？ → 想法。
- **概念与个人：** 您会在专业演讲中分享吗？ → 概念。这是私人反思吗？ → 个人。
- **想法与项目：** 有人正在研究它吗？是 → 项目。没有→想法。毕业的时刻就是工作开始的时刻。
- **写作与媒体：**写作是*神器*（论文）。媒体是*生产和分发基础设施*（内容管道、社会监控）。
- **写作与概念：** 概念页被提炼出来（200 字的事实汇编）。论文是成熟的散文（论点、叙述、故事）。
- **个人与公司：** 是关于*他们作为人*吗？ → 人/。与*组织*有关吗？ → 公司/。两个页面相互链接。
- **家庭与个人：** PA 会执行吗？ → 家庭（可操作）。这是私人反思吗？ → 个人。
- **来源与 .raw/ sidecars：** 每个实体的丰富数据 → .raw/ sidecar。批量多实体导入 → 来源/。
如果没有合适的内容，请在收件箱/中归档并标记它。这是模式需要发展的信号。

### 人员页面上的认知规则
背景部分（信念、动机、沟通方式、评估）是系统中价值最高的部分，但也是最容易产生幻觉的部分。智能体可能会根据稀疏的证据进行过度概括，或者过度拟合最近的一次交互。规则：
- **每项主张都引用其来源。** 不是“她咄咄逼人”，而是“她在 3 月 15 日的会议上强烈反对定价（观察）。”
- **三种来源类型：** `观察到的`（您看到它发生），`自我描述`（他们对自己说的），`推断`（您在字里行间阅读）。每一个都贴上标签。
- **信心跟踪互动次数。** 一次会议 = 低信心。五次会议=高。不要根据薄弱的数据编写明确的评估。
- **新近度很重要。** 2 年前的信念可能不是现在的。标记日期。
- **永远不要从单个数据点进行概括。** “她在一次会议上似乎很沮丧”是一个时间线条目。模式需要多次观察。
- **用户的更正优先于一切。** 如果用户说“她的说法是错误的”，请立即更新 - 该更正是系统中最高置信度的信号。

## 浓缩管道
**这是最重要的操作模式。** 每次您的座席在会议、电子邮件、推文、日历事件、联系人同步中遇到一个人或公司时，它都应该丰富相应的大脑页面。
充实不仅仅是“查找他们的 LinkedIn”。它是：
- **他们相信什么** — 立场、世界观、公共立场
- **他们正在建造什么** — 当前的项目，正在发货的东西
- **他们的动力** — 抱负、职业轨迹
- **他们的沟通方式** - 他们如何参与，什么激励他们
- **他们与你的关系** — 历史、背景、开放话题
- **确凿的事实** — 角色、公司、联系信息、资金（赌注）
事实是赌注。上下文就是价值。
### 何时丰富
**任何时间**出现个人或公司信号：
- 会议记录中提到某人 → 丰富
- 有人给你发电子邮件 → 丰富
- 有人在社交媒体上与你互动 → 丰富
- 出现新联系人→丰富
- 你在谈话中提到某人，而他们的页面很薄→丰富
- 一家公司宣布融资、发布产品、发布新闻→丰富内容
### 富集来源（按价值顺序）
1. **你自己的互动** - 你对他们说了些什么，他们对你说了些什么（最高信号）
2. **会议记录**——最丰富的上下文源
3. **电子邮件主题** — 语气、紧迫性、关系动态
4. **社交媒体**——信仰、公共定位、与谁互动
5. **网络搜索** — 背景、新闻、谈话
6. **人员 API** — 结构化档案数据（职业历史、教育背景、技能、联系信息）
7. **公司 API** — 融资、投资者、估值、员工人数、财务状况
8. **联系数据** — 电子邮件、电话、位置
### 数据源技巧
每个外部数据源都应该有自己的命名技能，并具有完整的 API 文档、身份验证模式和使用说明。丰富技能对它们进行编排——它根据层决定要调用“哪些”源，然后委托个人技能“如何”调用 API。
这使事情保持干燥：丰富技能拥有逻辑（何时丰富、什么层、提取什么），并且每个数据源技能拥有 API 契约（端点、身份验证、速率限制、陷阱、验证规则）。
推荐的数据源技能：
- **网络搜索** — 广泛的关键字搜索（Brave、Google 等）。快速的背景、新闻、资金。
- **语义搜索** — 比关键字更适合查找特定人员、LinkedIn URL、个人写作。 （例如，困惑等）
- **社交搜索** — X/Twitter、Bluesky 等用于公众声音：信仰、项目、参与模式。
- **人才丰富** — 类似 LinkedIn 的结构化数据：职业历史、教育背景、技能、联系信息。 （Crustdata、Proxycurl、人员数据实验室等）
- **网络搜索** — 搜索您的专业网络以获取热情的介绍和联系。 （偶发、粘土等）
- **公司情报** — 宣传册级数据：融资轮次、投资者、估值、员工人数、财务状况。 （Captain API、Crunchbase 等）
- **会议历史记录** — 搜索过去的会议以查找与此实体的交互。 （环背鱼、水獭、萤火虫等）
- **联系数据** — 您的联系人的电子邮件、电话、位置。 （谷歌通讯录等）
新人的典型充实流程：
1. **网络搜索** → 查找 LinkedIn URL、职业弧线、备用名称
2. **人才丰富** → 深层结构化数据（技能、工作经历、教育背景、联系信息）
3. **语义搜索** → 查找揭示信仰和观点的个人网站、谈话、写作
4. **社交搜索** → 他们的公众声音、他们与谁互动、爱好马
5. **网络搜索** → 新闻报道、最近新闻、谈话
6. **会议历史** → 过去与您的互动
对于新公司：
1. **公司情报** → 资金、投资者、员工人数、财务状况
2. **网络搜索** → 产品、新闻、牵引力
3. **社交搜索** → 公司的公开定位
4. **人员充实** → 充实创始人/关键团队成员（每个人都会触发人员充实）
### 丰富层（不要过度丰富）
- **第 1 层（关键人物）：** 完整渠道 - 所有来源。核心圈子、商业伙伴、重要合作者。
- **第 2 层（值得注意）：** 网络搜索 + 社交 + 大脑交叉引用。偶尔与您交往的人。
- **第 3 层（次要提及）：** 仅从源中提取信号，附加到时间线。其他每个人都值得追踪。
包含真实交互数据的瘦页面比填充通用 Web 结果的胖页面更好。不要将 10 次 API 调用浪费在没有公开露面的人身上。
### 原始数据 sidecar
每个丰富 API 响应都会保存为 JSON sidecar：
```
people/jane-doe.md              ← brain page (curated, readable)
people/.raw/jane-doe.json       ← raw API responses
```

JSON 由带有获取时间戳的源作为键控：
```json
{
  "sources": {
    "crustdata": { "fetched_at": "2026-04-05T...", "data": { ... } },
    "web_search": { "fetched_at": "...", "data": { ... } }
  }
}
```

大脑页面是精炼版本。原始数据是存档。
大脑页面中包含的内容（精炼）：位置、当前头衔、公司、标题、教育程度（一行）、职业弧线（精简）、顶级技能、社交句柄、个人资料图片永久链接。
仅保留在 .raw/ 中的内容：包含职位描述的完整工作历史记录、完整的技能列表、每个雇主的公司描述、特定于平台的 ID、关注者/连接计数、完整的 API 响应正文。
重新丰富时：用新数据+新时间戳覆盖源密钥。不要追加——替换。
### 验证规则
当从人员/公司 API 自动丰富时：
- **连接/关注者数量低（例如，<20）：** 可能是错误的人。使用 `"validation": "low_connections"` 标志保存到 .raw/。不要自动写入大脑页面。
- **名称不匹配：** 如果返回的名称与实体的姓氏不同，则跳过。
- **明显的笑话简介：** 职业弧提到荒谬的标题 - 跳过。
- **如有疑问：** 保存原始数据但不更新大脑页面。错误的数据比没有数据更糟糕。
### 浏览器预算
如果丰富涉及基于浏览器的查找（LinkedIn、经过身份验证的页面），请设置每日预算（例如，每天 20 次查找）以避免帐户标记。更喜欢基于 API 的丰富服务来进行批量工作 - 它们不会影响用户的浏览器会话。
## 进入标准 — 谁获得页面
并不是每个人都值得拥有大脑页面。根据关系重要性缩放页面创建：
**始终为以下内容创建页面：**
- 与您进行过一对一或小组会议的任何人
- 主要同事、合作伙伴和直接合作者
- 任何有牢固或更好工作关系的人
- 家人、亲密朋友、核心圈子
**如果信号存在则创建：**
- 最近互动过的联系人
- 在对话中提及名字的任何人
- 具有多个共享事件的事件联系人
**不要创建：**
- 群体活动嘉宾名单中的随机姓名，无需互动
- 没有识别上下文的单名条目
- 完全没有关系信号的联系人
当有疑问时：用户是否从现有的条目中受益？如果没有，请跳过它。
## 技能架构
技能是系统的模块化构建块。共有三种类型，了解它们的组成方式至关重要。
### 1.数据源技巧（叶子节点）
每个外部 API 或数据源都有自己的命名技能。该技能拥有 API 合约：端点、身份验证、速率限制、错误处理、验证规则以及响应的外观。
示例：
- **人员充实**（Crustdata、Proxycurl、人员数据实验室）——类似 LinkedIn 的结构化数据
- **网络搜索** (Happenstance、Clay) — 搜索专业网络，寻找相互联系
- **公司情报**（Captain API/Pitchbook、Crunchbase）——资金、投资者、财务状况
- **语义搜索**（Exa、Perplexity）—查找 LinkedIn URL、个人网站、写作
- **会议历史**（Circleback、Otter、Fireflies）——过去的会议记录和笔记
- **日历/联系人**（Google 日历、通过集成工具的 Google 联系人）— 日程、联系信息
- **社交媒体**（X API、Bluesky API）——公共帖子、参与度、关注者数据
- **工作区工具**（Gmail、Slack、Drive via 集成工具）— 电子邮件线程、消息、文档
数据源技能**永远不会由用户直接调用。**它们由编排技能调用（如下）。
### 2.编排技巧（协调员）
这些技能包含*逻辑* - 他们决定做什么，然后委托给数据源技能来了解如何去做。
**丰富技能**是最重要的编排技能。它决定：
- 这是创建（新页面）还是更新（新信号）？
- 该实体属于第几层？ （决定调用哪些数据源）
- 从源材料中提取哪些信号类型？
- 以什么顺序调用哪些数据源技能？
- 如何将结果写入大脑？
其他编排技巧：
- **会议摄取** — 从会议工具中提取会议，通过分析创建大脑会议页面，然后为讨论的每个与会者和公司调用丰富内容
- **电子邮件分类/行政助理** - 处理收件箱，处理日程安排，然后在遇到人员或公司时致电丰富
- **社交监控** - 扫描公共社交媒体以获取提及和参与度，然后调用丰富的知名帐户
### 3. 管道技能（端到端工作流程）
这些是将多种编排和数据源技能链接在一起的面向用户的技能：
- **早间简报** — 阅读日历 + 任务 + 大脑状态 + 最近的信号 → 生成简报
- **个人研究** — 给出一个名字，运行完整的第 1 层丰富并呈现结果
- **每周大脑维护** - 运行 lint、标记过时的页面、建议丰富目标
### 他们是如何组成的
```
User says "tell me about Jane Doe"
  → Agent searches brain (grep/index)
  → Page is thin → calls enrich skill (orchestration)
    → enrich determines Tier 1
    → calls happenstance skill (data source) → gets LinkedIn URL
    → calls crustdata skill (data source) → gets full profile
    → calls exa skill (data source) → finds personal writing
    → calls web_search (built-in tool) → gets press coverage
    → calls meeting history (data source) → finds past meetings
    → writes brain page, saves .raw/ sidecar, cross-references
  → Agent presents the enriched page to user
```

```
Cron fires "meeting ingestion" every afternoon
  → meeting-ingestion skill (orchestration) pulls new meetings
  → for each meeting: creates brain meeting page
  → for each attendee: calls enrich skill (orchestration)
    → enrich calls relevant data source skills based on tier
  → for each company discussed: calls enrich skill
  → extracts tasks, commits brain repo
```

关键见解：**数据源技能是无状态且可重用的。**丰富技能可以调用 Crustdata，无论触发器是会议、电子邮件、社交提及还是直接用户请求。数据源技能并不关心请求来自哪里。
## 如何将电线丰富到一切
丰富技能是中心枢纽。每个摄取路径都集中在它上面：
```
Meeting ingestion ───────┬─────────────────────────┬─── people enrichment API
Email triage ────────────┤                         ├─── company intelligence API
Social monitoring ───────┤    ENRICH SKILL         ├─── network search API
Contact sync ────────────┤   (orchestration)       ├─── semantic search API
Manual conversation ─────┤                         ├─── social search API
Calendar events ─────────┤                         ├─── web search
Webhooks ────────────────┴─────────────────────────┴─── meeting history API
                              │
                              ▼
                         BRAIN REPO
                    (people/, companies/,
                     meetings/, deals/)
```

每个进入丰富技能的箭头都带有一个**信号**（来自源的原始信息）和一个**实体**（要丰富的个人或公司）。丰富技能：
1. **检查大脑状态** - 页面是否存在？薄吗？
2. **确定层** — 第 1 层（完整管道）、第 2 层（网络 + 社交 + 交叉引用）、第 3 层（仅源提取）
3. **从源材料中提取信号**（信念、动机、轨迹、事实）
4. **基于层调用数据源技能**（每个技能都是一个命名的、记录的模块）
5. **写入大脑** — 创建（通过 RESOLVER.md）或更新（附加时间线，更新编译的事实）
6. **交叉引用** — 更新所有链接的实体页面
7. **将原始数据**保存到 `.raw/` sidecar
8. **提交**到大脑存储库
关键接线规则：**每个摄取技能都必须调用丰富。**这不是可选的或理想的。这是结构性的。如果添加新的摄取途径（例如 Slack 监控技能），其实施必须包括“对于提到的每个人/公司，调用丰富技能”。如果这条线缺失，大脑就会停止从该来源进行复合。
## 自动 Cron 作业
当你积极使用大脑时，大脑不会仅仅生长。 Cron 作业使系统变得自治——大脑得到维护，收件箱得到分类，会议得到吸收，提及内容甚至在你睡觉时也会受到监控。
### cron 架构
Cron 作业作为**独立的代理会话**运行 - 它们获得自己的上下文，读取自己的技能，并且不会阻塞主对话线程。他们可以发布到特定的通知渠道（Telegram 主题、Slack 渠道、Discord 线程）或静默工作。
每个 cron 作业本质上都是：“醒来，阅读一项技能，完成工作，发布结果（如果没有发生，则保持沉默），然后回去睡觉。”
### 为大脑驱动的系统推荐的 cron 作业
**高频（每 10-30 分钟一次）：**
- **电子邮件监控** — 扫描收件箱，按优先级分类，将摘要发布到通知渠道。直接处理低风险项目（日程安排、致谢）。
- **消息监控** — 检查关键通信渠道中是否有来自重要联系人的未回复消息。用建议的回应来呈现它们。
**中频（每 1-3 小时一次）：**
- **社交雷达** — 扫描公共社交媒体以查找对您或您的组织的提及、参与度、新兴叙事。对需要注意的项目发出警报。致电 rich 寻找与您互动的知名新客户。
- **心跳** — 综合检查。日历展望、任务回顾、电子邮件扫描、大脑状态回顾。如果有什么需要注意的地方就发帖；如果没有，请保持沉默。
**每日：**
- **早间简报** — 日历 + 任务 + 紧急事项 + 隔夜信号 → 一条通知。 “这是你的一天”的信息。
- **任务准备** — 归档昨天已完成的任务，根据日历 + 待办事项 + 经常性项目构建今天的列表。
- **会议摄取** — 从会议工具中提取所有新会议，运行完整摄取（创建会议页面、传播到实体页面、提取任务）。这是最繁重的 cron 工作——它涉及最多的大脑页面。
- **社交媒体收藏** — 存档您自己的帖子、跟踪参与速度、检测删除。输入媒体/页面。
**每周：**
- **Brain lint** — 运行完整的维护过程：矛盾、过时的页面、孤立的页面、缺少交叉引用、MECE 归档违规。发表报告。
- **丰富扫描** — 查找 90 多天内尚未丰富的大脑页面，或包含许多“[尚无数据]”部分的页面。将它们排队等待重新浓缩。
- **联系人同步** — 从您的联系人中提取最近添加的内容，与大脑交叉引用。为重要的新联系人创建页面。
### crons 如何喂养大脑
关键见解：**cron 作业是自主丰富引擎。**没有它们，大脑只有在您积极与代理交谈时才会增长。和他们在一起：
- 邮件监控遇到一个人→通话充实→大脑成长
- 会议摄取处理文字记录 → 为每位与会者丰富内容 → 大脑成长
- 社交雷达检测到一个新的知名账户 → 通话丰富 → 大脑成长
- 联系人同步找到新联系人→通话丰富→大脑成长
- 丰富扫描发现陈旧页面 → 使用新数据调用丰富 → 大脑保持最新状态
大脑复合 24/7 因为 cron 作业被连接来调用丰富。用户睡觉；大脑则不然。
### Cron作业设计规则
1. **当没有发生任何事情时保持沉默。** 如果 cron 没有发现任何新内容，则它不应该产生任何输出。没有“没有什么可报告”的消息。这很关键——嘈杂的 cron 会被禁用。
2. **发布到特定渠道。** 每个 cron 都会发布到其指定的通知渠道（例如，电子邮件 cron → 电子邮件主题、社交雷达 → 社交警报主题）。不要混合信号。
3. **为繁重的工作生成子代理。** cron 会话应该保持轻量级。如果会议摄取需要处理 5 个会议并更新 30 个实体页面，请生成用于实体传播的子代理。
4. **幂等和检查点感知。** 每个 cron 都应该跟踪它已经处理的内容（在像 `meeting-notes-state.json` 这样的状态文件中），这样它就不会在下次运行时重做工作。
5. **尊重安静的时间。** 除非事情确实紧急，否则不要在晚上 11 点到早上 7 点之间发帖。 crons 应该在发布之前检查时间。
6. **每个摄取cron都必须调用enrich。** 这是结构规则。处理会议但不丰富与会者的 cron 是一个错误，而不是一个功能。
### 示例：这一切如何组合在一起
自主大脑系统中的一个典型下午：
1. **下午 3:00** — 电子邮件监控 cron 触发。扫描收件箱。找到 3 封新电子邮件：日程安排请求、融资公告和创始人寻求建议。
   - 直接处理日程安排请求（检查日历，回复可用时间）
   - 在融资公告中对公司进行了丰富的呼吁 → 更新公司页面以进行新一轮融资
   - 将创始人的电子邮件发布到通知渠道以供用户处理
2. **下午 3:15** — 满足摄取 cron 火灾。查找今天开始的 2 个新会议。
   - 创建 2 个大脑会议页面并进行分析
   - 两次会议的通话丰富了 8 名与会者 → 更新了 8 人页面
   - 讨论了 3 个公司的通话内容丰富 → 更新了 3 个公司页面
   - 提取 4 个行动项目 → 添加到任务列表
3. **下午 3:30** — 社交雷达 cron 启动。检测到记者正在撰写有关用户组织的帖子。
   - 将警报发布到社交警报频道
   - 给记者打电话丰富 → 创建/更新他们的人员页面以及最近的活动
4. **下午 4:00** — 心跳触发。日历显示 1 小时后的会议。与会者的大脑页面最后一次丰富是在 3 个月前。
   - 给与会者带来新的丰富体验
   - 发布准备说明：“1 小时后与 X 会面。这是自上次见面以来发生的变化。”
用户没有要求任何这些。大脑增长了 12 页，用户在下午 4:00 的会议上做好了充分准备——因为管道连接正确。
## 来自生产系统的工作示例
这些示例展示了该架构如何端到端运行。名称和细节是通用的，但技能链是精确的——每个技能调用、每个文件写入、每个 cron 触发器都是它实际工作的方式。
### 示例 1：会议摄取 — 完整链
每天下午 3:00 会触发一个 cron 作业，并提示：“阅读技能/会议摄取/SKILL.md 并处理今天的会议。”
**第 1 步：技能链加载。** 会议摄取技能的序言中写着“阅读 Skills/enrich/SKILL.md”，因此代理会在接触任何数据之前加载丰富协议。这一点至关重要：这意味着代理知道如何处理遇到的每个人和公司。
**步骤 2：拉取新会议。** 座席调用会议历史数据源技能（在本系统中为 Circleback）。它检查跟踪上次处理的会议 ID 的状态文件（“memory/meeting-notes-state.json”）。自上次运行以来查找 2 个新会议。
**第 3 步：流程会议 1 —“与 Sarah Chen 和 Mike Torres 一起进行产品审查。”**
代理使用以下内容创建“brain/meetings/2026-04-07-product-review.md”：
- 它自己的在线分析（不是人工智能摘要的副本 - 通过大脑已经了解的参与者和项目进行重构）
- 关键决策、行动项目以及与其他大脑页面的连接
- 线下完整成绩单
**第 4 步：丰富与会者。**
对于 **Sarah Chen** — 代理搜索大脑：`grep -rl "Sarah Chen" /data/brain/people/`。找到 `people/sarah-chen.md`。读它。页面最后一次丰富是在两周前，并且覆盖面广。 → **第 3 层**：仅从本次会议中提取信号。在她的时间表中添加：“2026-04-07 | 会议 - 推迟发布时间表，需要更多 QA。担心 API 稳定性。”用新的后续项目更新她的开放线程。
对于 **Mike Torres** — 大脑搜索会找到 `people/mike-torres.md`。页面存在但很薄：只有名称、标题和之前的一个会议条目。 → **第 2 层**：网络搜索 + 社交 + 大脑交叉引用。特工找到他最近的博客文章（反馈到“他们相信什么”）、他的 X 活动（反馈到“Hobby Horses”），并将他与其他两个提到他的大脑页面进行交叉引用。更新了上面的编译事实。
对于**“来自 Meridian Labs 的 Alex”**（在会议中提到，但不是与会者）——大脑搜索什么也没发现。 → **创建路径**：
1. 读取 RESOLVER.md：“特定命名的人” → `people/`
2. 使用 schema.md 中的 person 模板创建 `people/alex-rivera.md`
3. 运行**第 1 层丰富**（完整管道）：网络搜索 → 查找 LinkedIn URL。人员丰富 API → 完整的结构化配置文件。语义搜索 → 查找会议演讲。网络搜索 → 找到有关 Meridian Labs 最近融资的新闻报道。
4. 将原始 API 响应保存到 `people/.raw/alex-rivera.json`
5. 交叉引用：更新“companies/meridian-labs.md”以链接到 Alex 的页面
**第五步：丰富讨论的公司。** Meridian Labs 得到了广泛的讨论。代理检查“companies/meridian-labs.md” — 是否存在，但资金数据已过时 4 个月。调用公司情报API → 获取最新的轮次数据。更新页面。
**第 6 步：提取操作项。** 在记录中找到 3 个操作项 → 附加到 `ops/tasks.md`。

```

## 摄取工作流程
这些是特定的摄取模式。每个人都将丰富称为其最终步骤。
### 会议摄取
每次会议结束后（通过 Circleback、Otter、Fireflies 或手动注释）：
1. 提取会议记录+完整记录
2. 创建一个大脑会议页面，其中包含**您自己的分析**（不仅仅是重复的人工智能摘要）——通过您对与会者世界的了解来重新构建
3. **传播到实体页面** - 为讨论的每个人和公司调用丰富。在更新实体页面之前，会议不会被完全吸收。
4. 将操作项提取到任务列表
5. 承诺
### 电子邮件摄取
处理电子邮件时：
- 提取提到的人和公司
- 通过电子邮件上下文（语气、请求、关系信号）丰富通话内容
- 记录日程安排、承诺、后续行动
### 社交媒体摄取
监控社交媒体时：
- 捕捉您跟踪的人公开发表的言论（信念、项目、观点）
- 检测参与模式（谁在回复你，谁在放大你）
- 为著名帐户致电丰富→输入“他们相信什么”和“Hobby Horses”部分
### 手动摄取
当你在谈话中提到某人或某事时：
- 你自己的评论是最高价值的信号——始终捕捉这些
- “在技术方面非常敏锐，可以成为基础设施项目的优秀顾问” → 立即出现在该人的页面中
- 如果大脑页面较薄，则触发全面浓缩
## 导航和并发
**index.md** — 内容目录。每一页都列出一行摘要。对于导航和查询路由很有用。
**log.md** — 按时间顺序记录摄取和更新。仅追加。
大规模（500 多页）添加搜索工具（嵌入、BM25 或 gbrain 等工具）。在中等规模下，grep 效果很好。
### 写热点和并发
一旦你的 cron 作业、摄取作业和子代理都触及大脑存储库，**index.md 和 log.md 就会成为合并冲突磁铁。** 每个工作流程都希望在每次提交时追加到 log.md 并更新 index.md。
实际缓解措施：
- **将 index.md 视为派生的，而不是手动维护的。** 不要在每个摄取工作流程中更新它，而是通过扫描目录树定期（每天或按需）重建它。这消除了它作为写入热点的情况。
- **使 log.md 附加安全。** 每个条目都是一个带有时间戳前缀的独立行。并发追加到文件末尾很少发生冲突。如果这样做，双方都是正确的——只要保留两条线即可。
- **批量提交，而不是按页面提交。** 当摄取作业更新 10 个实体页面时，最后提交一次，而不是 10 次。这减少了冲突面。
- **在推送之前拉动。** 每个工作流程都应该在推送之前`git pull --rebase`。通过仅附加日志和独立实体页面，变基几乎总是自动解析。
- **实体页面很少发生冲突。** 两个工作流程同时更新 `people/jane-doe.md` 的情况很少见，因为它们是由有关不同人的不同信号触发的。真正的冲突热点是共享文件（index.md、log.md），这就是为什么这些文件应该是仅附加的或派生的。
## 维护（Lint）
代理应该定期（每周）：
- **重复数据删除扫描：** 查找潜在的重复页面 - 不同页面上相似的名称、相同的公司、相同的电子邮件。确认后合并。
- **矛盾：** 检查页面之间的冲突事实（例如，两个页面列出同一公司同一个人的不同角色）。
- **过时：** 标志状态部分被较新的时间线条目取代。
- **孤立：** 查找没有入站链接的页面。
- **开放线程：** 检查似乎已解决但未移至时间轴的项目。
- **缺少交叉引用：** 实体 A 提及实体 B，但未链接到其页面。
- **缺少页面：** 经常提到但缺少自己的页面的实体。
- **MECE 归档：** 标记似乎位于错误目录中的任何页面。
- **来源审核：** 检查人员页面中高价值部分（信念、动机、评估）中是否有无来源声明。标记没有来源类型或日期的声明。
- **别名覆盖范围：** 检查最近的会议记录或电子邮件是否包含任何页面别名字段中尚未包含的名称变体。
## 这与 RAG 有何不同
RAG 在每个查询上从头开始重新获取知识。大脑预先计算合成并保持最新状态。具体来说：
- **交叉引用是预先构建的。** 您不需要法学硕士即可发现人员 A 在 B 公司工作并且参加了会议 C — 已链接。
- **预先标记矛盾。** 当新数据与旧数据冲突时，代理会在摄取期间（而不是查询时）解决或标记它。
- **编译是持久的。**摄入的每一个来源都会使大脑更加丰富。没有任何东西被丢弃或重新衍生。
- **结构本身就是一个提示。** 空部分（“他们相信什么：[尚无数据]”）告诉代理下一步要寻找什么。
## 页面生命周期
大脑页面可以具有隐式生命周期状态：
- **活跃：** 当前、最近更新、持续的关系或相关性
- **休眠：** 6 个多月内未更新，关系冷却，但仍有潜在相关性
- **存档：** 移至“存档/” — 倒闭的公司、结束的关系、解决的交易。仅历史记录。
- **毕业：** 对于成为项目的想法，或成为计划的项目 - 旧页面链接到新页面
在 lint 过程中，标记 6 个多月内未更新的页面以供审核。有些应该存档；其他人只需要一张新的丰富通行证。
## 是什么造就了一个伟大的大脑
一个伟大的大脑可以让你在参加任何会议、电话或决策时已经知道：
1. 这个人是谁以及他们关心什么（阅读 30 秒）
2. 公司的实际情况（不是他们6个月前说的）
3. 你们之间存在哪些开放话题（承诺、后续行动、交易）
4.最近发生了什么变化（最新的时间线条目）
5. 需要注意什么（模式、关注点、机会）
糟糕的大脑就是一堆没人读的 LinkedIn 内容和会议记录。一个好的大脑是编译环境的，可以让你在每次互动中更加有效。
## 解析器
创建或归档新页面时，请遍历此决策树。每一种知识都有一个归宿。
### 决策树
**从这里开始：主要主题是什么？**
1. **特定的指定人物** → `people/`
2. **特定组织**（公司、基金、非营利组织、政府机构）→ `公司/`
3. **金融交易**，包含条款和决策 → `交易/`
4. **在特定时间发生的特定会议/通话的记录 → `会议/`
5. **正在积极构建的东西**（有存储库、规范、团队或活跃的工作）→ `projects/`
6. **尚未有人构建的原始可能性** → `ideas/`
7. **关于世界如何运作的可重复使用的心智模型或论文** → `概念/`
8. **一篇散文**，可以作为独立作品出版 → `writing/`
9. **您机构的战略、组织、流程、内部动态** → `org/`
10. **政治或公民景观** — 政策、立法、选举、政府 → `公民/`
11. **公共叙事或内容运营** — 社会监控、内容管道、发布的帖子 → `媒体/`
12. **重大人生计划**——包含多个项目的持久承诺领域→“计划/”
13. **国内运营** — 物业、物流、家务管理→`家务/`
14. **私人笔记** — 健康、个人反思、内心生活 → `个人/`
15. **招聘渠道** — 候选人评估、角色规格、面试笔记→“招聘/”
16. **可重复使用的 LLM 提示** — 用于从模型获取特定输出的模板 → `prompts/`
17. **原始数据导入或快照** — 批量导出、API 转储、定期捕获 → `sources/`
18. **代理可交付成果** — 由您的代理制作的简报、摘要和研究 → `代理/`
19. **未排序/快速捕获** - 你还不知道它去了哪里 → `inbox/`
20. **死亡/不再相关** - 没有活跃引用的历史页面 → `archive/`
### 消歧规则
当两个目录看起来合适时，应用这些决胜局：
- **个人与公司：** 如果页面是关于*他们作为一个人*（信仰、关系、轨迹），那么它就是人/。如果是关于*他们运营的组织*，那就是公司/。两个页面相互链接。
- **概念与想法：** 你能把它作为一个框架“教”给某人吗？概念。你能“建造”它吗？主意。
- **概念与个人：** 您会在专业演讲中分享吗？概念。这是私人反思吗？个人的。
- **想法与项目：** 有人正在研究它吗？如果是，则进行项目。如果没有，想法。毕业的时刻就是工作开始的时刻。
- **写作与概念：** 概念是经过提炼的（200 个单词的汇编事实）。写作是成熟的散文（论点、叙述、故事）。
- **写作与媒体：**写作是*神器*。媒体是*生产和发行基础设施*。
- **组织与计划：** org/ 是*关于*您的组织的机构知识。计划/是关于您的个人角色和其中的优先事项。
- **公民与人民：**政治人物获得人民/页面。他们的立法议程和作为公民行为者的政治定位属于公民/。
- **家庭与个人：** 如果 PA 对其执行，则它是家庭（可操作）。如果它是私人反思，那么它就是个人的（内心生活）。
- **来源与 .raw/ sidecars：** 每个实体的丰富数据 → 实体旁边的 .raw/ sidecar。批量多实体导入 → 来源/。
- **代理与来源：**来源*进入*大脑。代理可交付成果是合成的输出，可馈送到*您的阅读中*。
###特殊目录（不是知识）
这些存在于大脑存储库中，但不是知识目录：
- **templates/** — 每种类型的页面模板（结构性的，而非内容性的）
- **attachments/** — 二进制附件（图像、PDF）。由您的编辑管理，而不是由代理人管理。
### MECE 检查
每一条知识都应该经过上面的决策树并准确地落在一个目录中。如果您发现确实不适合任何类别的内容，请将其归档到 inbox/ 并标记它 - 这是模式需要发展的信号。
＃＃ 入门
1. 创建上面的目录结构（或让您的代理创建）
2.为每个目录编写一个`RESOLVER.md`决策树和一个`README.md`解析器
3. 使用页面约定和模板编写“schema.md”
4. 将大脑规则添加到代理的配置（AGENTS.md 或等效文件）中作为硬规则
5. 从一份会议记录或您想要跟踪的一个人开始
6. 让代理构建前几页，检查它们，并迭代架构
7. 连接您的会议工具以触发摄取
8. 连接浓缩以激发每个新人/公司的信号
9. 大脑从那里合成
人类的工作：整理资源、直接分析、提出好的问题，并思考这一切的含义。代理人的工作：其他一切。
