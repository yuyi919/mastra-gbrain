# Phase 3.1 Learnings: 全量测试和类型检查

## Decisions

### 补充缺失的 `content_hash` 类型约束
在测试与底层存储插入页面的时候，强制补全了 `content_hash` 参数。

**Rationale:** 因为新的接口 `PageInput` 要求此参数为必填，旧测试未提供导致 `tsc` 类型不匹配。
**Source:** `03.1-full-test-typecheck/03.1-01-SUMMARY.md`

---

## Lessons

### 深入 Effect Error 类型的处理机制
Effect 库中的 `catch` 处理中如果抛出未知的错误，必须用 `SqlError` / `SchemaError` 严格包裹。原生 `catch (e) => e` 会破坏类型签名。

**Context:** 在 `UnsafeSql.ts` 以及事务包裹中解决 `Effect.tryPromise` 抛出的联合错误不符合 `EngineEffect` 的问题。
**Source:** `03.1-full-test-typecheck/03.1-01-SUMMARY.md`

---

## Patterns

### 强类型检查的防线 (`npx tsc --noEmit`)
在执行 `bun test` 后，必须伴随严格的 TypeScript 类型构建检查，这比单纯依赖单测更加稳健。

**When to use:** 当进行基础接口 (Store Service) 大幅重构，且应用了复杂的泛型、联合类型推导（如 Effect）时，单测未必能覆盖所有的类型退化情况。
**Source:** `03.1-full-test-typecheck/03.1-01-SUMMARY.md`

---

## Surprises

### 错误的分支历史覆盖
在执行计划时，由于 `git` 环境指向了一个 detached 或者不正确的 fetch 分支，引发了之前规划文件丢失和代码回归。

**Impact:** 中断了自动化流程，必须手动寻找并拉取 GitHub 上的远程 PR 分支。这警示在 `gsd` 重度依赖 Git 历史的工作流中，分支的健康和对齐是先决条件。
**Source:** `.planning/ROADMAP.md` 和会话反馈
