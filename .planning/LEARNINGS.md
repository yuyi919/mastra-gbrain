# LEARNINGS

## Effect 迁移：保持上层接口不变
在引入 Effect 运行时时，优先保证对外接口签名与调用方式稳定，通过在适配层做 Effect↔Promise 桥接实现渐进迁移，避免引发上层大面积联动重构。

## 重构前置：用测试覆盖率驱动推进
在替换底层运行时与查询构建前，先把测试覆盖面梳理清楚，用测试作为回归护栏推进重构，能显著降低“替换成功但功能退化未察觉”的风险。

## 大型类重构：方法映射表先行
对遗留大类做替换时，先建立“旧方法→新实现”的映射字典（表格化），用于追踪进度、对齐行为语义、减少遗漏与重复实现。

## 事务：回滚与边界是首要风险点
在把事务逻辑迁移到 Effect / Drizzle 组合后，事务回滚与同步/异步边界更容易出现盲区；必须靠集成测试覆盖事务失败路径，避免在“看似成功”的 happy path 下埋雷。

## 类型系统：环境组合与错误类型会“爆炸式扩散”
多个 Effect 环境（如 SqlClient、Store Service）叠加时，漏传环境或签名不一致会触发大规模类型错误链；通过明确的类型别名与统一的错误封装，可以显著改善可读性与排错效率。

## SQLite 兼容：避免依赖非标准语法扩展
不要假设 SQLite 引擎支持某些非标准扩展（例如 UPDATE ... LIMIT）；在需要“限制更新范围”时改用子查询/CTE 等兼容写法，避免运行时异常。

## 测试隔离：连接释放不彻底会制造幽灵锁
集成测试或并行单测如果没有严格释放连接/关闭资源，容易出现文件锁定与环境残留，进而产生虚假的 busy/locked 报错；测试基建必须强制 dispose 并确保临时 DB 路径隔离。

## Git：慎用 reset --hard，分支健康是自动化前置条件
`git reset --hard` 会无条件丢弃本地工作，和自动化工作流叠加时风险极高；同时在执行规划/执行命令前必须确认分支与工作区健康，避免在 detached 或错误分支上导致计划与成果丢失。

## Effect 错误处理：不要让未知错误“泄露”进类型签名
Effect 的错误通道需要结构化且可预测：捕获未知错误时应包裹为显式的 SqlError/SchemaError 等受控错误类型，避免用“原样抛出”的方式破坏错误联合类型与上层可组合性。
