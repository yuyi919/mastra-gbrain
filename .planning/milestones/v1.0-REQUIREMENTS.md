# 需求文档 (Requirements)

## 1. 用户故事与目标 (User Stories & Goals)
**目标**：将 `LibSQLStore` 的底层实现替换为经过 `effect` 重构的 `BrainStore` 运行时。

**作为一名开发者/系统架构师**，我希望底层存储层被 `effect` 函数式运行时驱动，以便我能够获得更强大且可组合的错误处理、依赖注入以及并发控制能力，从而提升代码库的稳定性。
**作为一名终端用户**，我希望整个重构过程对外部不可见，即所有的 AI Agent 检索、页面导入和后台清理脚本（如 `doctor`，`backlinks`，`embed`）都能按预期稳定运行。

## 2. 核心功能需求 (Functional Requirements)
1. **引入 BrainStore 运行时**：将当前系统中基于 `bun:sqlite` 或 `drizzle-orm` 的异步（Promise-based）操作迁移到 `Effect` 运行时的封装中，特别是利用 `@tslibs/effect` / `@effect/sql-drizzle` 等生态。
2. **替换 `LibSQLStore` 内部实现**：
   - 现有的 `LibSQLStore` 需充当 `BrainStore` 的适配器，确保对外的 `GBrainStore` 等接口签名不变。
   - 重构 `getHealthReport`, `getStaleChunks`, `upsertVectors`, `markChunksEmbedded`, `searchKeyword`, `hybridSearch` 等核心逻辑的底层访问层。
3. **完善错误处理与类型安全**：
   - 使用 `Effect` 的类型系统消除现存的 `any` 泛滥与跳过类型检查（如 `@ts-expect-error`），确保通过 `tsc --noEmit` 检查。
   - 所有数据库异常需在 Effect 管道中被妥善处理，并映射为统一的错误模型。
4. **推进下一步 `effect` 重构**：
   - 完成本次替换后，输出针对更上层（Agent, Workflow, Search）的 `effect` 重构可行性分析报告或后续重构指南。

## 3. 非功能性需求 (Non-Functional Requirements)
- **向下兼容性 (Backward Compatibility)**：所有现存的 `test/*.test.ts` 测试用例，尤其是与 `libsql` 和运维脚本相关的集成测试，必须 100% 绿色通过，保证系统回归测试无故障。
- **性能指标 (Performance)**：新的 `BrainStore` 运行时在批处理（Batch Insert, Chunk Upsert）场景下不能有明显的性能退化，需验证原有 N+1 插入瓶颈是否有所改善。
- **环境隔离 (Environment Isolation)**：所有测试必须通过 `StoreProvider` 和 `EmbeddingProvider` 依赖注入传入独立的 `file::memory:` 或本地 SQLite 测试数据库，且测试后必须通过 `dispose()` 释放资源。

## 4. 约束条件 (Constraints & Constraints)
- **技术栈限制**：仅限使用 Bun 作为运行时，TypeScript 作为语言，`drizzle-orm` 作为对象关系映射工具，底层数据库必须兼容 libSQL (支持向量和全文检索)。
- **开发规范**：每次完成特定阶段后，必须使用 `git commit` 并附带语义化的提交信息。绝不向主分支提交破坏现有功能的半成品代码。
