# GSD 异常处置手册

本页不解释理想流程，而是解释流程跑偏时怎么收拾局面。

## 1. 我不知道该用哪个命令

先不要猜。直接把目的说出来：

```bash
$gsd-do 修复登录按钮
$gsd-do 我想开始一个新里程碑
```

`$gsd-do` 的价值不是帮你干活，而是帮你把自然语言路由到正确流程。它适合“我知道目标，但不确定入口命令”的时刻。

## 2. 我中断了工作，回来时上下文丢了

优先使用恢复流，不要靠翻聊天记录。

```bash
$gsd-progress
$gsd-resume-work
```

如果是刻意暂停，应该在离开前先做：

```bash
$gsd-pause-work
```

这样 GSD 会留下接力点，而不是让你下次重新回忆。

## 3. 计划写出来了，但我怀疑方向错了

不要直接执行。先检查假设：

```bash
$gsd-list-phase-assumptions N
```

如果问题是“需求理解可能不一致”，回到：

```bash
$gsd-discuss-phase N
```

如果问题是“技术路线可能选错”，回到：

```bash
$gsd-research-phase N
```

先修正输入，再重做计划，比执行后返工便宜得多。

## 4. 执行中发现任务其实不属于当前阶段

根据影响范围分流：

- 不影响路线图，只是小插单：用 `$gsd-quick` 或 todo
- 影响当前里程碑顺序：用 `$gsd-insert-phase`
- 是未来某轮才该做的想法：用 `$gsd-plant-seed`

`$gsd-plant-seed` 适合“现在不做，但以后某个条件满足时必须想起来”的事项，比普通待办更适合长期埋点。

## 5. quick task 变成了真正的项目工作

不要因为已经写了 `.planning/quick/` 就继续把它当 quick。出现下面任一条件，就应该升级：

- 它开始改变里程碑优先级
- 它需要多轮计划和验证
- 它明显不再是“顺手处理”
- 它需要在 `ROADMAP.md` 里被看见

升级方式通常是：

1. 把 quick task 产出的结论抽出来
2. 新增阶段或插入阶段
3. 在阶段计划里把 quick 期间形成的认识正式化

quick 的目标是减少流程成本，不是隐藏主线工作。

## 6. 验证没过，但代码已经写完

这是正常情况，不是流程失败。正确处理顺序是：

1. 读 `VERIFICATION.md` 或 UAT 输出
2. 区分“真缺口”与“人工待确认项”
3. 真缺口回到执行流修补
4. 人工待确认项明确标记，不要伪装成已验证

如果你使用的是 quick full 或阶段验证流，宁可让状态停在 `Needs Review` / `Gaps`，也不要口头宣布“差不多好了”。

## 7. 里程碑完成后，文档还是很多，看起来很乱

这是因为 GSD 默认更重视可追溯，而不是自动帮你删历史。应对方式不是删文件，而是区分文档身份：

- 仍在指导当前工作的：保留在顶层 `.planning/`
- 已经属于历史快照的：进入 `.planning/milestones/`
- 只是在当前区里占地方的旧阶段目录：用 `$gsd-cleanup` 挪走

如果你真正的问题是“哪些该留，哪些该归档”，直接看 [document-lifecycle.md](./document-lifecycle.md)。

## 8. 想做发布，但不想把 `.planning/` 一起带进 PR

这时不要手工清理提交历史。优先用：

```bash
$gsd-pr-branch
```

它的作用是把代码提交从 planning 提交里分离出来，给 PR 一个干净分支。不要再用破坏性的 git 操作去“顺手整理历史”。

## 9. 我想保留 GSD 流程，但不想把 planning 文档提交进仓库

通过 `.planning/config.json` 控制：

```json
{
  "planning": {
    "commit_docs": false,
    "search_gitignored": true
  }
}
```

这样规划文档仍然能用，但不需要纳入版本库。适合客户项目、公开仓库或不希望暴露内部规划的场景。
